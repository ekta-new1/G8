1. CP Mindset (Competitive Programming Mindset)
The CP mindset refers to the way of thinking and problem-solving approach that competitive programmers develop to solve coding problems quickly and efficiently.

Key Aspects of CP Mindset:

Problem Understanding:
Carefully read and understand what the problem asks for. Clarify inputs, outputs, and constraints.

Breaking Down Problems:
Divide complex problems into smaller, manageable parts or steps. Think about the best way to approach each part.

Optimized Thinking:
Always think about making your solutions faster (more efficient in terms of time and memory).

Practice & Repetition:
Regularly solve different problems to improve skills. Experience teaches you shortcuts and tricks.

Learning Data Structures & Algorithms:
Knowing common tools like arrays, trees, graphs, recursion, dynamic programming, etc., helps to solve problems faster.

Debugging & Testing:
Always test your code thoroughly and fix bugs quickly.

Time Management:
In contests, time is limited. Practice teaches you to solve problems efficiently within the time frame.

Stay Calm & Focused:
Keep cool during contests. Don't panic if stuck; try to analyze and find a different approach.

Big Picture:
Having a CP mindset means approaching problems with logic, efficiency, continuous learning, and confidence. This mindset helps in performing well in competitive programming contests.

2. Time Complexity
Time complexity measures how the running time of an algorithm increases as the size of the input data (n) increases.
It helps us understand how fast or slow an algorithm is, especially for large inputs.

Common Notations:

O(1) - Constant Time:
The algorithm takes the same time regardless of input size. The time taken remains the same no matter how much data exists.
Example: Accessing a specific element in an array.

int[] arr = {10, 20, 30, 40, 50};
System.out.println(arr[2]);

Whether the array has:
5 elements
500 elements
50,000 elements

Accessing arr[2] always takes almost the same time.

Real-life Example:
Finding the 3rd book on a shelf when you already know its exact position.

O(log n) - Logarithmic Time:
Time increases slowly as input grows.
Example: Binary search in a sorted array.

Instead of checking one by one, we repeatedly divide the data into halves.

Find 75 in:

1 2 3 ... 100

Steps:

Check middle (50)
75 > 50, search right half
Check middle (75)
Found!

Only a few checks are needed.

Real-life Example:
Finding a word in a dictionary by opening near the middle and repeatedly narrowing down.

O(n) - Linear Time:
Time increases directly with the number of elements.
Example: Looping through all elements once.

for(int i=0; i<n; i++) {
    System.out.println(arr[i]);
}

If:

n = 10 → 10 operations
n = 100 → 100 operations
n = 1000 → 1000 operations

Real-life Example:
Checking attendance of every student in a class.

O(n log n) :
Slightly slower than linear, common in efficient sorting algorithms like merge sort and quicksort.
it's actually a combination of O(n) and O(log n).

Let's take Merge Sort as an example.

int arr[] = {8, 4, 2, 6, 1, 9, 3, 5};

Step 1: Divide into halves (Log n part)

Merge Sort keeps dividing the array into 2 halves:

8 4 2 6 1 9 3 5
      ↓
8 4 2 6      1 9 3 5
   ↓             ↓
8 4   2 6    1 9   3 5
 ↓ ↓   ↓ ↓   ↓ ↓   ↓ ↓
8 4   2 6    1 9   3 5

For 8 elements:

8 → 4 → 2 → 1

It takes 3 levels to reach single elements.

Notice:

log₂(8) = 3

This is the log n part.

Step 2: Merge back (n part)

Now Merge Sort combines them:

8 + 4     → 4 8
2 + 6     → 2 6
1 + 9     → 1 9
3 + 5     → 3 5

Then:

4 8 + 2 6     → 2 4 6 8
1 9 + 3 5     → 1 3 5 9

Then:

2 4 6 8 + 1 3 5 9
→ 1 2 3 4 5 6 8 9

At every level, we touch all n elements.

This is the n part.

O(n^2) - Quadratic Time:
Usually occurs when we have nested loops.

Example:
for(int i=0; i<n; i++) {
    for(int j=0; j<n; j++) {
        System.out.println(i + " " + j);
    }
}

If:

n = 10 → 100 operations
n = 100 → 10,000 operations
n = 1000 → 1,000,000 operations

Real-life Example:
Every student shaking hands with every other student.

O(2^n) - Exponential Time:
Very slow; doubles with each additional input.
Every time the input increases by 1, the work roughly doubles.

Example: Generating All Subsets

Suppose you have 3 items: A B C

Possible subsets:

{}
{A}
{B}
{C}
{A,B}
{A,C}
{B,C}
{A,B,C}

Total subsets = 8

2³ = 8

For 4 items:

2⁴ = 16 subsets

For 5 items:

2⁵ = 32 subsets

Every new item doubles the number of possibilities.

O(n!) - Factorial Time:
Extremely slow; used in some brute-force problems.
It occurs when an algorithm tries every possible arrangement (permutation) of items.
Notice how quickly it grows:
3!	6
4!	24
5!	120
6!	720
10!	36,28,800

Real-Life Example: Seating Students 

Suppose there are 3 students:

A B C

Possible seating arrangements:

ABC
ACB
BAC
BCA
CAB
CBA

Total arrangements:

3! = 6

For 4 students:

4! = 24 arrangements

For 5 students:

5! = 120 arrangements

The algorithm must check every arrangement.

Understanding & Use:
Knowing the time complexity helps in choosing the right algorithm for large data sets. For example, avoid O(n^2) for very large n because it becomes slow.


| Big O      | Meaning           | Example                   |
| ---------- | ----------------- | ------------------------- |
| O(1)       | Constant Time     | Access array element      |
| O(log n)   | Logarithmic Time  | Binary Search             |
| O(n)       | Linear Time       | Traverse array            |
| O(n log n) | Linearithmic Time | Merge Sort                |
| O(n²)      | Quadratic Time    | Bubble Sort               |
| O(2ⁿ)      | Exponential Time  | Generate all subsets      |
| O(n!)      | Factorial Time    | Generate all permutations |


3. Big O Analysis -

Big O notation is the way to describe how the running time or space requirement of an algorithm grows with the size of the input (n).
It provides a high-level understanding of the efficiency of algorithms without getting lost in exact details.

It is a mathematical way to measure the efficiency of an algorithm by describing how its execution time or memory requirement grows as the input size increases.
